HackTheon 2025 Prelims Writeup

HackTheon Sejong is a Korean national-scale CTF competition held annually in Sejong City. It’s aimed at university students and early-career security folks, but the difficulty ramps up fast — especially in the later challenges. This year’s event had teams competing across forensics, reverse engineering, web, mobile, and misc categories.

This is a writeup of the challenges we managed to solve during the competition. The write-up covers what we looked at, what we tried that didn’t work, and eventually what got us the flag. Hopefully useful for anyone who hit the same walls we did.

Barcode

Overview:
We’re given a binary called barcode and a file called original-flag.barcode. The binary takes a hex string as input and outputs an ASCII barcode made of spaces and asterisks *. The goal is to figure out what hex input produces the barcode in the given file.

Initial Analysis

The first thing we do is just run the binary with some test input to see what comes out:

Output is a small ASCII barcode pattern using spaces and *. Each hex byte maps to a specific row pattern in the barcode. So the challenge essentially is: given the target barcode output, reverse it back to the original hex input.

We also run file and strings on the binary to get some basic intel before diving in:

Nothing too revealing from strings — no hardcoded keys or hints. It’s a compiled binary with fairly standard structure. We skip deep static analysis for now and just work empirically, running it with known inputs to map out the encoding.

Looking at the pattern, a * represents a 1 bit and a space represents a 0 bit. We can read each row of the barcode as a binary number and convert it back to hex. Simple enough on paper.

First Approach — Direct Bit Conversion

We write a quick function to convert a barcode row back to its hex byte:

The string is reversed before converting because the barcode encodes bits in reverse order. Running this on the first few rows of original-flag.barcode gets us the character F — which is the start of FLAG{. Promising.

So the naive plan is: read every row from the barcode file, convert each one back to a hex byte, concatenate, and we have the flag.

But it doesn’t work for the rest of the bytes.

The Dependency Problem

After getting F cleanly, the next byte comes out wrong no matter what we try. After staring at the binary output more carefully, it becomes clear that the barcode for each byte isn’t purely derived from that byte alone — the output for byte N is influenced by byte N-1. There’s some kind of carry or state being passed between bytes in the encoding.

We confirmed this by running the binary with two different payloads that share the same second byte but differ in the first:

The output for the second byte is different in both cases even though the byte itself is the same. That confirms it — the encoding is stateful, so we can’t decode each byte independently.

Rather than spending hours reversing the algorithm to figure out the exact state machine, we decide to brute force it. Since each byte is only 8 bits, there are only 256 possible values per position. We can just try all of them and check which one produces a barcode row that matches the target.

Brute Force Approach

The strategy:

  1. Keep the known correct prefix fixed.
  2. For each unknown byte position, try all 256 possible hex values.
  3. Run the binary with the current payload and compare the output against the target barcode row.
  4. When we find a match, lock that byte in and move to the next.

The key observation is that original-flag.barcode gives us the ground truth to compare against. We know exactly what the correct output should look like at every row.

Here’s the script we used:

The script iterates through all 256 possible byte values for each unknown position, runs the binary, and checks whether the output matches the expected barcode row. We repeat this process byte by byte until the full hex string is recovered.

It’s not elegant but it works. We run the script, watch it cycle through candidates, and when a row matches we lock that byte and move on. The whole thing takes a couple of minutes to run through each unknown position.

FLAG{1325ed439e52bba40fedefaf5bec9458}

Why This Works

The JavascriptInterface bridge is a well-known attack surface in Android apps. When developers expose native methods to JavaScript without validating what URLs the WebView can load, any page loaded in the WebView — including attacker-controlled pages — can call those native methods. In this case the decode function does exactly what we want, we just had to make it call it.

This vulnerability is covered under CWE-749 — exposed dangerous method or function. In a real application this could let an attacker read sensitive data, call internal APIs, or exfiltrate credentials. The proper fix is to either restrict which origins the WebView can load (using shouldOverrideUrlLoading) or avoid exposing sensitive native methods to JavaScript altogether.

Also worth noting — the @JavascriptInterface annotation was introduced in Android 4.2 specifically because before that, all public methods of an injected object were accessible from JavaScript, not just the ones you intended to expose. The annotation narrows it down, but it only helps if you’re careful about what you annotate. In this challenge the developer annotated the decode function without thinking about who could call it.

Cat

Overview:
We’re given a hash and told to crack it. The challenge is called Cat — as in hashcat.

Identifying the Hash

The hash we’re given looks like a SHA-256 hash based on its length (64 hex characters). In hashcat’s mode numbering, SHA-256 is -m 1400.

Building the Mask

The key part of this challenge is figuring out the right mask for the brute force. Hashcat uses mask-based attacks (-a 3) where you specify the character set and length pattern for candidate passwords.

The mask we used:

Breaking it down:

  • ?l — lowercase letter
  • ?d — digit
  • ! — literal exclamation mark
  • So the pattern is: letter digit letter letter letter digit ! digit digit

This is likely based on a hint from the challenge or a known password policy. The total keyspace for this mask is manageable for hashcat.

Running Hashcat

Flags:

  • m 1400 — SHA-256 mode
  • a 3 — mask attack
  • o cracked.txt — output the cracked password to a file

Hashcat churns through the keyspace and finds the match. The cracked password is the flag.

FLAG: h4ckm3!25

Notes on the Mask

If you’re not familiar with hashcat mask syntax, here’s a quick breakdown of the charset placeholders:

Placeholder Charset
?l Lowercase letters (a-z)
?u Uppercase letters (A-Z)
?d Digits (0-9)
?s Special characters
?a All of the above

Literal characters like ! are just typed directly into the mask. So our mask ?l?d?l?l?l?d!?d?d describes a 9-character password following a specific pattern. The total keyspace is 26 × 10 × 26 × 26 × 26 × 10 × 1 × 10 × 10 which is around 45 billion combinations — sounds like a lot but hashcat burns through SHA-256 at hundreds of millions of hashes per second on a decent GPU, so this cracks in minutes.

Watch

Overview:
We’re given RDP bitmap cache files from a system. The challenge name “Watch” refers to someone watching activity on a machine over RDP. We need to reconstruct what was visible on screen during that session.

What is an RDP Bitmap Cache?

When you connect to a machine over RDP, Windows caches bitmap tiles of the remote screen locally to avoid retransmitting parts of the screen that haven’t changed. These cache files get stored on the client machine at:

The cache files are named things like Cache0000.bin and bcache24.bmc. They contain raw pixel data split into small tiles, but they’re not stored in any standard image format — they’re in a proprietary binary format.

The challenge gives us exactly these files: Cache0000.bin and bcache24.bmc.

The .bin files hold the main persistent cache — these survive across sessions. The .bmc file is the “bitmap cache” in the older format. Both can be processed with the same tool.

Before jumping straight to extraction, we do a quick check on the files:

The hex dump shows a repeating structure of fixed-size chunks, which confirms it’s the tile-based bitmap cache format. Each tile is a fixed pixel dimension (typically 64x64 or 32x32 depending on the RDP negotiation) stored as raw RGB data.

Extracting the Cache with bmc-tools

bmc-tools is a forensic tool developed by ANSSI (the French national cybersecurity agency) specifically for parsing and extracting data from RDP bitmap cache files. It’s the go-to tool for this kind of challenge.

We run it against our cache file:

Flags:

  • s — source cache file
  • d — output directory for the extracted bitmaps
  • b — output as a single stitched collage image

The -b flag is important here. Instead of dumping hundreds of individual tile PNG files, it assembles them into a single large collage image. This makes it much easier to visually scan for readable content without having to manually sort through hundreds of fragments.

Reading the Output

Once the collage is generated, we open it and scan through. RDP bitmap caches tend to contain fragments of whatever was on the screen — file explorers, text editors, terminals, browser windows, whatever the user had open. The tiles are stored in order of last use, so recently used screen areas tend to be well-represented.

In our collage, we can see fragments of a text editor with the flag written in it. The tile stitching gives us enough context to read it cleanly.

A few things to keep in mind when analyzing these outputs:

  • Not every tile will be readable — some are UI chrome, window decorations, or solid-color backgrounds
  • Text is often split across tile boundaries, so you need the stitched view rather than individual tiles
  • If the flag isn’t immediately obvious in the b collage, try dumping individual tiles with just s and sorting them by size or content

FLAG{s0m3on3_1s_w4tch1n9_my_pc}

Nothing is Essential

Overview:
We’re given a forensic disk image — DISK.ad1. The .ad1 format is an AccessData proprietary format, which means FTK Imager is the tool of choice for mounting and browsing it. The challenge hints at finding something that “isn’t essential” — likely pointing at a note or memo that wouldn’t normally be considered critical evidence.

Mounting the Image

We open DISK.ad1 in FTK Imager using File > Add Evidence Item > Image File. FTK Imager mounts the image and gives us a full directory tree to browse.

The image contains a Windows installation. We navigate to the user directory and export the john7 user profile for further analysis.

FTK Imager also lets you preview files in-app without exporting, which is useful for quick triage. We use this to quickly scan file types and modification timestamps before committing to a full export.
